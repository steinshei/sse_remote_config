# SSE 容量模型、监控指标与 SLO 验收（与 OKR 对齐）

**状态**: 设计基线（上线前需用压测与线上观测校准常数）  
**对齐文档**: `App自研远程配置库研发计划.md` §2、§4、§7、§8；`docs/api-contract-v1.md`（SSE 路径与参数）

本文档补充可行性分析中「观测与 SLO」「SSE 容量模型」条目：给出**可计算的粗算模型**、**建议采集的指标**，以及如何把 §8 **OKR** 落成 **SLI / SLO** 与验收方式。

## 1. 目标读者与范围

- **读者**: 后端 / SRE / 客户端负责人（联调观测）。
- **范围**: SSE 长连接集群、Redis Pub/Sub 扇出、与配置 REST（`/config/latest`、`/config/delta`）共用入口时的容量与稳定性；不含业务运营后台 UI。

## 2. SSE 容量模型（粗算）

### 2.1 符号与输入

| 符号 | 含义 | 典型取值来源 |
|------|------|----------------|
| \(D\) | 日活（DAU） | 产品数据 |
| \(r\) | 峰值在线率：峰值同时在线 / DAU | 一般可先取 **0.15～0.35**（强留存产品可能更高） |
| \(s\) | 前台活跃客户端中「应维持 SSE」的比例 | 架构上若后台断开 SSE，\(s\) 会小于 1 |
| \(C\) | 目标峰值并发 SSE 连接数 | \(C \approx D \times r \times s\) |
| \(N\) | SSE 网关/API 实例数 | 部署拓扑 |
| \(c_\text{max}\) | 单机「安全」并发 SSE 上限 | 压测 + 80% 以内作为规划值 |

**规划连接数（保守）**:

\[
C_\text{plan} = D \times r_\text{high} \times s_\text{max}
\]

例：\(D=200\) 万，\(r=0.25\)，\(s=0.9\) → \(C\approx 45\) 万并发；若单机验证 \(c_\text{max}=8\) 万（仅示例），则至少 **6 台** 专司 SSE，并预留 **≥30%** 余量应对营销活动与版本日尖峰。

### 2.2 内存与句柄（单机）

对每台实例做**上限预算**（需在预发用 `ss -s`、`/proc/sys/fs/file-max`、应用堆外缓冲验证）：

- **每连接内存**：事件循环中的 socket 缓冲、TLS、应用层 Session 对象；经验上常落在 **几十 KB～数百 KB/连接**（与是否透传 body、是否缓存最近发布消息等相关）。
- **文件描述符**：每连接至少 1 FD；加上 DB/Redis/日志，单机 FD 预算应 **远大于** \(c_\text{max}\)（例如 2×～3×）。

**验收动作**: 在预发打满 \(0.8 \times c_\text{max}\) 连接并保持 30 分钟，观察 RSS 增长、GC（若适用）、Full GC pause、502/连接重置率。

### 2.3 Redis Pub/Sub 扇出

设一次配置发布需要向 **所有订阅该 app/环境 channel 的 SSE worker** 推送通知：

- **发布频率** \(f\)：运营发布 + 自动化回滚（次/分钟级通常不高，但要预留突发）。
- **消息体积** \(m\)：仅通知「有新版本」应尽量小（版本号、etag）；避免全量配置塞进 Pub/Sub。

**风险点**: Pub/Sub 在集群模式下订阅与发布不一致、或单 channel 订阅者极多时的 **CPU 与网络突发**。模型上要求：

- 发布路径**异步化**（Worker 池），避免阻塞 HTTP 主线程。
- 对「全局大广播」做 rate limit 或分片 channel（如按 `app_id` + `shard`）。

### 2.4 负载均衡与 idle 超时

SSE 为**长连接**：LB / 反向代理 / 云 WAF 的 **idle read/write 超时** 必须大于服务端 **心跳注释（comment）间隔** 的计划值（参见 `api-contract-v1` 心跳约定），否则会出现「无业务发布但大量静默断开」。

**容量相关验收**: 在网络路径上故意植入 NAT/LB 超时（或等价缩短），验证客户端指数退避与 delta 补偿是否将「配置陈旧时间」压在 SLO 内。

### 2.5 压测清单（最小集）

| 项 | 通过标准（示例，与下文章 SLO 绑定） |
|----|-------------------------------------|
| 阶梯升压至目标 \(C_\text{plan}\) | 错误率、P99 延迟无持续恶化；无 OOM |
| 保持连接 1h + 周期性 comment 心跳 | 无 mass disconnect |
| 模拟单实例下线（优雅 / 不优雅） | 重连风暴后系统恢复；无版本倒退 |
| 大发布（高扇出） | Pub/Sub 与 CPU spike 在阈值内；REST 仍可用 |

## 3. 监控指标（建议最小集）

指标命名可按 Prometheus 风格；关键是**服务端 + 客户端 + 发布链路**三段齐全。

### 3.1 服务端（SSE / API）

| 指标域 | 指标含义 | 用途 |
|--------|-----------|------|
| 连接 | `sse_active_connections`（按 app、环境、实例） | 容量趋势、扩缩容 |
| 吞吐 | `sse_connect_total`, `sse_disconnect_total`（附 reason 标签） | 抖动、重连风暴 |
| 延迟 | `config_fetch_duration_seconds`（latest/delta 分桶） | 与「实时性」OKR 同源 |
| 错误 | HTTP 5xx 率、`sse_upstream_errors_total` | 稳定性 |
| 发布 | `config_publish_total`, `pubsub_publish_latency_seconds` | 扇出瓶颈 |
| 资源 | CPU、内存、FD、GC、event loop lag（Node） | 单机上限验证 |

### 3.2 客户端（SDK 埋点）

| 指标域 | 事件 / 字段 | 用途 |
|--------|----------------|------|
| fetch | 成功/失败、耗时、HTTP 状态、`since_version` | 拉取稳定性、补偿路径 |
| SSE | 连接建立耗时、断线原因、重连次数、comment 收到间隔 | 连接健康、与心跳/LB 对齐 |
| 生效 | 「自获知新版本 → 本地 active 生效」耗时（若可测） | **P99 < 5s** 的直接证据之一 |
| 安全 | 沙盒失败、延迟生效跳过、崩溃回滚触发 | 安全性 OKR |

### 3.3 合成探测（可选）

- 自地区向外对 **`/config/latest` 与 SSE 握手** 做黑盒探测，区分「业务逻辑错误」与「链路不可用」。

## 4. SLO、错误预算与 OKR 对齐

研发计划 §8 **成功指标（OKR）** 在此映射为可验收的 **SLI / SLO**。窗口建议：**滚动 30 天**（与月度复盘一致）；重大活动周可缩短评审周期。

### 4.1 稳定性：配置下发成功率 > 99.9%

| 项 | 定义 |
|----|------|
| **SLI** | 在客户端视角，一次「应成功的配置获取」中，是否成功得到可解析的响应并最终进入可用状态（含 delta 合并成功）。 |
| **分母** | 冷启动触发的 fetch + 轮询兜底 fetch +（可选）SSE 通知后触发的 fetch；去重规则需在实现期写死（如「同版本跳过」是否计入分母）。 |
| **SLO** | 月度可用性 ≥ **99.9%**（等价最大允许失败约 **43.2 分钟/月** 若按恒定请求率近似）。 |
| **验收** | 报表：SLI 比值；按 `app_version`、`region` 维度下钻；**事故单**关联时间段应有可解释的 5xx/超时 spike。 |

**错误预算用尽时**: 冻结非必要发布、扩容/降级 SSE、启用更保守超时与熔断。

### 4.2 实时性：SSE 在线时 P99 生效延迟 < 5 秒

| 项 | 定义 |
|----|------|
| **SLI** | \(t_\text{visible} - t_\text{publish}\)：服务端接受**生效**配置的时间戳到客户端**读到新 active** 的时间差；仅统计 SSE 连接为 **OPEN** 的会话。 |
| **SLO** | 该延迟的 **P99 < 5s**（滚动 30 天）。 |
| **验收** | 需**可对照时钟**（服务端 NTP、客户端允许 ±1s 误差）或基于同一条 `version` 号在日志中关联；MVP 可用采样 + 端到端测试环境探针证明，再扩到全量。 |

**补充**: SSE 不在线时，以轮询/delta 路径满足的延迟可单独定义**次级 SLO**（例如 P99 < 5 分钟），避免与在线路径混算。

### 4.3 安全性：配置导致崩溃率 < 0.01%

| 项 | 定义 |
|----|------|
| **SLI** | 归因于「配置变更相关」的崩溃 / 会话数；分母为活跃会话或启动次数（需与现有的崩溃上报口径一致）。 |
| **SLO** | **< 0.01%**（滚动 30 天，按自然日聚合可外加「单日不高于 3× 基线」护栏）。 |
| **验收** | 崩溃栈与「最近生效配置版本」关联规则需在客户端打标签；**误报控制**见研发计划 §3.4（仅高风险 key 或归因信号触发回滚）。 |

### 4.4 效率：运营发版时间（定性 → 可测）

| 项 | 定义 |
|----|------|
| **SLI** | 自运营在后台点击「发布」到 **99% 目标用户**可在拉取路径拿到的最短墙钟时间（与灰度策略绑定）。 |
| **SLO** | 从「小时级」降到「分钟级」：建议落地为 **P95 < 15 分钟**（可按组织习惯调整），需与金丝雀百分比与分地区灰度方案一致。 |
| **验收** | 后台审计日志时间戳 + 客户端版本覆盖报表联合出图。 |

### 4.5 成本：低于 Firebase 同用量成本 50%

| 项 | 定义 |
|----|------|
| **SLI** | 月度云费用 + 专属人力（可选折算）与 Firebase Remote Config 账单（同 DAU/请求量估算）对比。 |
| **SLO** | **≤ 50%**（财务季度复盘）。 |
| **验收** | 成本归属标签（计算/带宽/日志/DB）；与 §2 容量模型一起评审，避免 SSE 规模上去后成本反转。 |

## 5. OKR ↔ 指标一览（汇总表）

| OKR（§8） | SLI 核心 | 主要数据源 | SLO / 验收 |
|-----------|-----------|------------|------------|
| 配置下发成功率 | fetch 成功比 | 客户端埋点 + 服务端 2xx/4xx/5xx | ≥ 99.9% / 30d |
| 配置生效延迟 | 发布到生效 P99 | 服务端 + 客户端关联 `version` | P99 < 5s（SSE 在线） |
| 配置导致崩溃率 | 归因崩溃率 | 崩溃平台 + 配置版本标签 | < 0.01% / 30d |
| 运营发版时间 | 发布到高覆盖 P95 | 审计日志 + 覆盖报表 | P95 < 15min（示例） |
| 基础设施成本 | 月度总成本 | 财务/云账单 | ≤ Firebase 估算的 50% |
